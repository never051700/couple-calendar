# 우리 캘린더 코드 평가

## 결론

- 최초 상태: **4.5/10 (C-)**
- 이번 보강 후: **7.0/10 (B- / 좋은 MVP, 운영 준비 중)**

구조와 읽기 쉬운 TypeScript는 괜찮은 MVP입니다. 이번 작업으로 iOS 설정 오류,
치명적인 RLS 우회, 푸시 토큰 노출·오귀속, 초대 재사용, 로그아웃 알림 잔존,
계정 삭제 부분 실패, 종일 일정과 한국어 날짜 문제를 보강했습니다. 다만 자동화
테스트 범위와 운영 관측성이 아직 부족해 바로 대규모 공개 서비스에 투입할 수준은
아닙니다.

## 항목별 평가

| 항목 | 점수 | 평가 |
| --- | ---: | --- |
| 구조/모듈화 | 7.5/10 | 화면, 컨텍스트, 데이터/알림 도우미 분리가 명확함 |
| 가독성/타입 | 7/10 | strict TypeScript와 공용 EventForm이 장점 |
| 기능 정확성 | 7/10 | 기존 일정 알림 동기화까지 보강했지만 시간대/장기 일정 경계 테스트가 더 필요 |
| 보안/개인정보 | 8/10 | 0002·0003 적용과 Edge Function 배포가 전제 |
| 테스트 | 3/10 | 핵심 날짜 단위 테스트만 존재, RLS/E2E/UI 테스트 없음 |
| 배포 준비 | 7/10 | iOS·Android 설정은 정리됐지만 외부 계정/실기기 검증이 남음 |

## 이번에 해결한 고위험 문제

- 본인 멤버십의 `space_id`/`role`을 바꿀 수 있던 직접 UPDATE 정책 제거
- 소유 이벤트를 관계없는 공간으로 옮길 수 있던 UPDATE 정책 수정
- 6자리 재사용 초대를 12자리 일회용 코드로 변경하고 두 명 제한 추가
- 파트너가 읽을 수 있던 프로필에서 푸시 토큰을 별도 다중 기기 테이블로 분리
- 구형 중복 토큰 폐기, 현재 기기의 원자적 토큰 재등록, 로그아웃 실패 재시도
- 웹훅 비밀값 검증과 실제 DB 레코드 재조회 추가
- 로그아웃 시 해당 계정의 푸시 토큰과 로컬 예약 알림 정리
- 중단에도 이전 값을 보존하는 iOS Keychain 이중 버퍼 세션 저장소 추가
- 가입/로그인 분리 및 앱 내 계정 영구 삭제 경로 추가
- 계정 삭제와 공유 캘린더 소유권 이전을 DB 트랜잭션으로 통합
- 앱 복귀 시 기존 Android/원격 수정 일정의 로컬 알림 동기화
- Expo SDK 56 설정 오류, iPhone safe area, 알림 탭 이동, 한국어 날짜 수정
- Android 런처·알림 아이콘, 정확한 일정 알람 권한과 알림 채널 수정

## 남은 우선순위

1. Supabase 로컬/스테이징 환경에서 RLS 통합 테스트 추가
2. 로그인 → 초대 → 실시간 CRUD → 알림 → 계정 삭제 E2E 테스트 추가
3. 캘린더 쿼리를 표시 기간 기준으로 제한하고 페이지네이션 적용
4. Sentry 같은 오류 수집과 구조화된 서버 로그 추가
5. 비밀번호 재설정 흐름 추가
6. 기존 데이터 점검 후 0002의 `NOT VALID` 제약을 `VALIDATE CONSTRAINT`로 확정
7. CI에서 `npm test`, `npm run check`, iOS·Android export를 필수 검사로 실행
8. Expo가 상위 전이 의존성의 npm 보안 경고를 해소하면 호환 SDK로 갱신
9. 실제 Android APK에서 FCM, 로컬 알림, 기존 서명키 업데이트 설치 검증

중요: 보안 점수는 `0002_ios_and_security_hardening.sql`,
`0003_atomic_account_deletion.sql`과 두 Edge Function이 실제 Supabase 프로젝트에
배포됐을 때의 평가입니다. 코드만 올리고 DB를 갱신하지 않으면 기존 취약점이 그대로
남습니다. 현재 `npm audit --omit=dev`의 중간 등급 경고 11개는 Expo의 빌드 도구
전이 의존성에 있으며, 제안된 Expo 46 강제 다운그레이드는 호환성을 깨므로 적용하지
않았습니다.
